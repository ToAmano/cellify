# GEMINI.md - 開発記録とコード規約

本ファイルは、AIアシスタント「Gemini」との共同開発によって設計・実装された `cellify` プロジェクトの開発経緯、技術選定、およびコード品質に関する規約をまとめた記録です。

---

## 1. プロジェクトの背景と経緯

### 開発の目的
DFT計算（VASP, Quantum ESPRESSO等）におけるスーパーセル作成および構造モデリング（ドーピング、空孔、表面スラブ）を、コマンドライン（CLI）から「手軽に」「パラメータを維持して」実行できるツールを提供すること。

### 決定プロセス
1.  **既存ツールの調査**: `cif2cell`, `phonopy`, `ASE`, `pymatgen` などの利点と欠点（インストール壊れやすさ、計算パラメータ引き継ぎの手間など）を分析。
2.  **要件定義（README.md）**: 「計算可能（Calculation-ready）な入力ファイルの生成（特にQEのnat自動更新など）」および「最小距離に基づく自動セル拡張」などを盛り込んだ要件を定義。
3.  **名称決定**: `scel` などの候補から、拡張性を考慮して **`cellify`** に決定。
4.  **GitHubリポジトリ**: `ToAmano/cellify` リポジトリをパブリックで作成・接続。

## 2. 開発・コーディング規約

### ① コード全体への型指定（Type Hints）の導入 【厳守】
本プロジェクトのコード（`src/cellify/` 配下のすべてのモジュール）は、保守性と可読性を向上させるため、**完全に型アノテーション（Type Hints）が施されています**。

*   **関数の引数・戻り値**: すべての関数およびメソッドに、`typing` モジュール等を用いた明確な型指定を付与します。
    ```python
    from typing import Tuple, Dict, Any
    from pymatgen.core import Structure

    def load_structure_file(filepath: str) -> Tuple[Structure, Dict[str, Any]]:
        ...
    ```
*   **ローカル変数**: 必要に応じて複雑な型を持つ変数にもアノテーションを記述します。
*   **静的解析の保証**: 今後コードを追加・修正する際も、型ヒントを記述し、`mypy` 等の静的型チェッカーでエラーがない状態を維持してください。

### ② 計算パラメータの保護（プレーンテキスト置換）
Quantum ESPRESSOなどのパラメータ・構造混在フォーマットにおいては、パースエラーや意図しないデフォルトパラメータ（例: `K_POINTS None`）の混入を防ぐため、元のテキスト記述（コメント、インデント）をプレーンテキストのまま保持し、`nat`/`ntyp`などの必要箇所のみを正規表現で書き換える「テキスト置換エンジン（`text_replace` モード）」を基本設計とします。

### ③ 拡張性のあるアーキテクチャ（I/Oアダプターとモデリングコアの分離）
将来の各種DFTソフトへの対応と機能拡張のため、以下の設計方針を維持します。
*   **adapters パッケージ**: 各計算ソフト特有のファイル読み込み・書き出し（パラメータ保護）ロジックは、抽象基底クラス `BaseAdapter` を継承した個別のアダプター（例: `EspressoAdapter`, `StandardAdapter`）にカプセル化します。
*   **core モジュール**: スーパーセル化や欠陥導入などの幾何学的モデリングは、I/Oを意識しない疎結合な設計（`Structure` を受け取り `Structure` を返すクリーンな関数群）を維持します。
*   **コアの細分化について**: 現状の規模ではファイルの見通しを良くするため単一の `core.py` に留めますが、機能数が増加した段階で速やかに `core/` パッケージへと分割可能なモジュール性を担保します。
